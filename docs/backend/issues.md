## Issue 2: TXT and PDF Ingestion Pipeline with Manual Trigger and Status

**What to build:**
Build the first complete ingestion path: load TXT and PDF files, chunk them (RecursiveCharacterTextSplitter, 512 chars / 64 overlap), call the OpenRouter embedding API, and write chunks into ChromaDB with `filename` and `file_type` metadata. Expose `POST /ingest` to trigger a run manually and `GET /ingest/status` to report doc count, last run timestamp, and last error. This is the vertical slice that proves the full ingest-to-store path works before adding more file types or automation.

**Acceptance criteria:**
- [ ] `POST /ingest` triggers ingestion of all TXT and PDF files in `WATCH_FOLDER` and returns HTTP 200 with a confirmation message.
- [ ] After ingestion, `GET /ingest/status` returns `{ "doc_count": <N>, "last_run_at": <timestamp>, "last_error": null }` where N equals the number of documents ingested.
- [ ] Calling `POST /ingest` a second time without changing files produces no duplicate chunks in ChromaDB.
- [ ] ChromaDB chunks for a TXT or PDF file each carry `filename` and `file_type` in their metadata.

## Issue 4: Image Ingestion with Vision Embedding

**What to build:**
Extend the ingestion pipeline to handle JPEG, JPG, and PNG files. Each image is loaded as raw bytes via Pillow and sent directly to the OpenRouter VL embedding model (no OCR, no captioning). ChromaDB stores only the `file_path` in metadata — no content field — so the image can be reloaded at query time. Each image produces exactly one chunk.

**Acceptance criteria:**
- [ ] A JPEG, JPG, or PNG file in `WATCH_FOLDER` is ingested and produces exactly one chunk in ChromaDB.
- [ ] The chunk's metadata contains `file_path` and `file_type: image`; the content field is absent or empty.
- [ ] The embedding for the image chunk is generated by sending raw image bytes to the OpenRouter VL model (not a text description).
- [ ] After image ingestion, `GET /ingest/status` reflects the updated doc count.

## Issue 5: Hash-Based Change Detection and Startup Scheduler

**What to build:**
Add SHA256-based change detection so the ingestion run knows which files are new, modified, or deleted — and acts on each accordingly (insert, re-ingest, or delete ChromaDB chunks). Wire this into an asyncio background task launched via FastAPI `lifespan` that runs ingestion immediately at startup and then every 60 seconds. `GET /health` is updated to reflect whether the initial startup ingestion has completed.

**Acceptance criteria:**
- [ ] A new file added to `WATCH_FOLDER` is ingested on the next scheduler run (within 60 seconds of being added).
- [ ] A modified file (hash changed) has its old ChromaDB chunks deleted and is re-ingested on the next run.
- [ ] A deleted file has its ChromaDB chunks removed on the next run.
- [ ] `GET /health` returns `"initial_ingestion_complete": true` after the startup ingestion run completes.
- [ ] The scheduler does not block the FastAPI event loop; queries can be served during ingestion.

## Issue 6: BM25 Index and Hybrid Retrieval with RRF Fusion

**What to build:**
Build an in-memory BM25 index from all text chunks in ChromaDB at startup, then wire it together with ChromaDB vector similarity into a `EnsembleRetriever`. Fuse the two result lists with Reciprocal Rank Fusion (`1 / (rank + 60)`), deduplicate, and return the top-3 final chunks. Images are excluded from BM25 but participate in vector search. Rebuild the BM25 index after each ingestion run so it stays current.

**Acceptance criteria:**
- [ ] A keyword query returns up to 5 BM25 candidates from text chunks only (no image chunks in BM25 results).
- [ ] A semantic query returns up to 5 vector search candidates (may include image chunks).
- [ ] The fused result contains at most 3 chunks after RRF re-ranking and deduplication.
- [ ] A term that appears verbatim in a document ranks higher after hybrid fusion than it would from vector search alone.
- [ ] After a new ingestion run, the BM25 index is rebuilt and newly ingested documents are findable by keyword.

**Blocked by:** Issue 2

## Issue 7: LangGraph RAG Pipeline (Retrieve → Relevance Check → Generate / No-Results)

**What to build:**
Implement the complete LangGraph pipeline: define `DocuFetchState`, implement four nodes (`retrieve`, `relevance_check`, `generate`, `no_results`), and assemble them into a compiled graph with a conditional edge that routes based on `is_relevant`. The `retrieve` node calls hybrid retrieval from Issue 6. Both `relevance_check` and `generate` call Claude Haiku via `ChatAnthropic`. Multi-turn history is accumulated in `messages`, trimmed to 20 before each LLM call.

**Acceptance criteria:**
- [ ] Invoking the graph with a query that matches ingested documents returns a 2–4 sentence answer with at least one source filename cited inline.
- [ ] Invoking the graph with a query that does not match any ingested document returns exactly `"I couldn't find relevant information in your documents."`.
- [ ] The `messages` list in state grows with each invocation and reflects prior turns in the conversation.
- [ ] History is trimmed to the last 20 messages before each LLM call; older messages are dropped.
- [ ] `sources` in the returned state contains only unique filenames of the chunks used in the answer.

**Blocked by:** Issue 6

## Issue 9: CLI Client

**What to build:**
Write `cli.py` — a standalone script that uses `httpx` to call the running FastAPI server. On startup it generates a UUID4 session ID and enters a prompt loop. Typed text goes to `POST /chat`; `/ingest` calls `POST /ingest`; `/status` calls `GET /ingest/status` and prints the result; `/quit` exits. If the server is unreachable, a clear error message is printed instead of a traceback.

**Acceptance criteria:**
- [ ] Running `cli.py` presents a `>` prompt; typing a question prints the answer and source filenames.
- [ ] The same session ID is used for the entire CLI session, enabling follow-up questions to reference prior answers.
- [ ] `/ingest` prints the server's confirmation message.
- [ ] `/status` prints doc count, last run time, and last error in a readable format.
- [ ] `/quit` exits the process cleanly.
- [ ] If the server is not running, the CLI prints a human-readable error (e.g. `"Server not reachable at http://localhost:8000"`) rather than a raw exception traceback.

**Blocked by:** Issue 5, Issue 8
